2. Hosting

The dashboard, the public dataset and the static documentation pages are served from the Hostinger production domain. ThreatPulse application-generated logs (managed-scheduler, cron jobs, JSONL logs, failed- job logs and verification logs) are written to the operator-configured log directory. The Hostinger managed scheduler runs the V6.2 portable jobs (refresh-dataset, refresh-baseline, publish, GC, verify, backup) and periodically contacts the documented public upstream sources (CISA KEV, NVD, FIRST EPSS, OSV, GitHub Advisory) from the operator's server. None of those third parties receive a visitor's IP address; only the server's IP is exposed during the upstream refresh.

ThreatPulse application-generated logs are retained for up to 30 days. Hostinger infrastructure / access logs are controlled by the hosting provider and are subject to Hostinger's applicable policies and configuration; ThreatPulse cannot delete Hostinger-controlled logs.

3. Data the application processes

ThreatPulse Radar is a public-data + local-only dashboard. The application processes the following categories of data:

  • Public vulnerability data (CVE records, EPSS scores, KEV entries, GitHub Advisory records, OSV records) — loaded from the operator's server-side refresh pipeline and served to every visitor from the operator's data root. No PII.
  • Local workspace, environment, remediation and report data — entered by the visitor in their own browser and stored in IndexedDB on the visitor's own device. Never uploaded.
  • Imported SBOMs and inventory files — read by the visitor's browser, parsed in a Web Worker, and stored in IndexedDB. Never uploaded.
  • Browser cache — a 1-hour TTL cache of the public dataset in localStorage, expired on manual refresh. No PII.
  • Server access logs — ordinary HTTP access logs (request path, response status, response size, user-agent) written by the ThreatPulse application and retained for up to 30 days. Hostinger infrastructure logs are out of scope for the application retention policy.

4. Cookies and tracking

No application-controlled cookies are set or read by ThreatPulse code. The full audit is in the cookie & storage notice and the audit script scripts/verify-v69-privacy-and-runtime-hardening.mjs. Because the application cannot observe hosting- or platform-level cookie behaviour, the operator must run the live verification checklist documented in the V6.9 audit record before declaring the deployment complete.

5. No account system

There is no sign-up, no sign-in, no profile and no cross-device sync. Every piece of data the visitor enters is stored locally on the visitor's own device.

6. Recipient categories

The public dataset is CORS-open. The recipient categories are:

  • Any HTTP client (browser, CI pipeline, security dashboard) that fetches /api/dataset or /.netlify/functions/dataset.
  • The operator's server-side refresh pipeline, which contacts CISA / NVD / FIRST EPSS / OSV / GitHub Advisory to populate the prebuilt dataset.

Local visitor data (workspace, environment, remediation, reports) is not shared with any third party because it never leaves the visitor's device.

7. Retention

ThreatPulse application and security logs are processed to maintain the security, availability and integrity of the service, detect abuse and diagnose technical errors. ThreatPulse application-generated logs are retained for up to 30 days.

Visitor-entered data (workspace / environment / remediation / reports) is retained in the visitor's own browser until the visitor explicitly clears site data or invokes one of the documented "Clear" actions in the dashboard. The dashboard never uploads this data to the server.

8. Local storage

Workspace, environment, remediation and report information is stored locally in the user's browser to provide functions requested by the user. ThreatPulse does not receive this locally stored information unless the user independently chooses to transmit an exported file.

9. Legal basis

ThreatPulse application and security logs are processed to maintain the security, availability and integrity of the service, detect abuse and diagnose technical errors. The intended legal basis is Article 6(1)(f) GDPR (legitimate interests of the controller), subject to final operator legal review.

The legal-basis language above is suitable for later legal review by the operator; it is not legal advice.

10. User rights

Because no visitor account exists and no visitor data is held on the server, the visitor's data-protection rights are exercised in the visitor's own browser through the documented "Export" and "Clear" actions on each local surface (workspace, environment, remediation, reports). The dashboard also surfaces the browser's native "Clear site data" / "Clear cookies and site data" controls for a full reset.
